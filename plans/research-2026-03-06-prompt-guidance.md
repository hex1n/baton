# Research: OpenAI GPT-5.4 Prompt Guidance 深度分析

> 来源: https://developers.openai.com/api/docs/guides/prompt-guidance

## 一、文章定位与背景

这不是一篇泛泛的"prompt 技巧"文章，而是一份**工业级的模型行为控制规范**。它回答的核心问题是：

> 如何通过结构化 prompt 让 LLM 在复杂、多步骤、工具密集的工作流中表现可靠？

GPT-5.4 被定位为平衡**长任务性能、行为控制、纪律执行**的主力模型。文章透露了一个重要信号：模型越强大，越需要精确的"行为合约"来约束它。

### GPT-5.4 擅长领域

- 长响应中保持人格和语调一致性
- Agentic 工作流的多步骤完成鲁棒性
- 长上下文或多工具场景中的证据丰富综合
- 模块化、基于技能的 prompt 中的指令遵循
- 大规模或多文档输入的长上下文分析
- 批量和并行工具调用的准确性
- 电子表格、金融、Excel 工作流的保真度和自验证

### 仍需显式引导的领域

- 会话早期的低上下文工具路由
- 需要前置检查的依赖感知工作流
- 基于任务形态的推理力度选择
- 需要纪律性来源收集的研究任务
- 需要执行前验证的高影响操作
- 需要清晰工具边界的编码和终端环境

## 二、核心设计哲学

### 2.1 合约式 Prompt（Contract-based Prompting）

文章最突出的理念是将 prompt 视为**合约**而非"提示"。几乎每个模式都以 `<xxx_contract>` 或 `<xxx_rules>` 的 XML 标签包裹，形成明确的行为规范：

- `<output_contract>` — 输出格式合约
- `<completeness_contract>` — 完成度合约
- `<verification_loop>` — 验证合约
- `<citation_rules>` — 引用合约
- `<grounding_rules>` — 事实锚定合约

与传统"请帮我做X"的 prompt 风格形成鲜明对比。**合约式设计的本质是把模糊的期望变成可验证的条件**。

### 2.2 防御性设计（Defensive Prompting）

多个模式在防御模型的已知弱点：

| 弱点 | 对应防御模式 | 机制 |
|------|------------|------|
| 空结果就放弃 | `<empty_result_recovery>` | 至少尝试1-2种备选策略 |
| 跳过前置步骤 | `<dependency_checks>` | 强制检查先决条件 |
| 过早停止 | `<completeness_contract>` | 未完成不得终止 |
| 缺少上下文就猜 | `<missing_context_gating>` | 不猜，查或问 |
| 高风险操作不验证 | `<verification_loop>` + `<action_safety>` | 飞行前/后检查 |

即使是 GPT-5.4 级别的模型，**可靠性仍然需要通过显式规则来保障**，而非依赖模型的"常识"。

### 2.3 推理努力作为调节旋钮（Reasoning as a Tuning Knob）

明确指出：**不要把 reasoning effort 当作改进的主要手段**。推荐的优先级：

```
更好的 prompt 结构 > 输出合约 > 验证循环 > 提高推理等级
```

推理等级建议：

| 等级 | 适用场景 |
|------|---------|
| `none` | 执行型任务（字段提取、分发路由、短结构化变换） |
| `low` | 需要少量推理的延迟敏感任务 |
| `medium` / `high` | 研究型任务（多文档综合、冲突解决、策略写作） |
| `xhigh` | 仅限长程 agentic 任务，**不推荐作为默认值** |

**算力不是万能的，结构化约束的 ROI 远高于暴力推理。**

## 三、关键 Prompt 模式详解

### 3.1 输出结构与效率

两个核心块：

**输出合约** — 精确返回请求的部分，按请求的顺序，如果要求特定格式则只输出该格式。

**冗余控制** — 偏好简洁、信息密集的写作；避免重复用户请求；保持进度更新简短；但不能过度缩短导致遗漏证据或推理。

### 3.2 默认跟进策略

```
意图清晰 + 可逆 + 低风险 → 直接执行，不必询问
不可逆 / 有外部副作用 / 缺少敏感信息 → 询问许可
执行后 → 简述做了什么 + 剩余可选项
```

这是一个**自主性阈值模型**：根据操作的风险等级动态调整是否需要人类确认。

### 3.3 指令优先级

```
用户指令 > 默认风格/语调/格式/主动性偏好
安全/诚实/隐私/权限约束 > 一切（不可被覆盖）
新指令 > 旧指令（冲突时）
不冲突的旧指令 → 保留
```

### 3.4 工具持久性与依赖管理

**工具持久性规则：**
- 只要工具能实质性改善正确性/完整性/锚定性，就使用
- 不要在另一个工具调用可能改善结果时提前停止
- 持续调用直到任务完成 + 验证通过
- 空结果或部分结果 → 换策略重试

**依赖检查：**
- 执行前检查是否需要前置发现/查找/记忆检索步骤
- 不因最终操作看起来明显就跳过前置步骤
- 如果任务依赖前一步输出 → 先解决依赖

### 3.5 并行 vs 顺序工具调用

```
独立查询 → 并行（减少等待时间）
有依赖关系 → 顺序（保证正确性）
并行检索后 → 暂停合成，再决定下一步
偏好选择性并行 → 并行化独立的证据收集，非投机性/冗余的工具使用
```

### 3.6 完成度合约

- 任务在所有请求项被覆盖或标记为 `[blocked]` 之前视为未完成
- 维护内部交付物清单
- 对列表/批次/分页结果：确定预期范围、跟踪已处理项、完成前确认覆盖率
- 被阻塞项 → 标记 `[blocked]` + 明确说明缺少什么

### 3.7 验证循环

四步验证：
1. **正确性**：输出是否满足每个要求？
2. **锚定性**：事实声明是否有上下文或工具输出支撑？
3. **格式性**：输出是否匹配要求的 schema 或样式？
4. **安全性**：下一步是否有外部副作用？有则先询问许可

**缺失上下文门控：**
- 缺少上下文 → 不猜
- 可检索 → 用工具查找
- 不可检索 → 问最小化的澄清问题
- 必须前进 → 显式标注假设 + 选择可逆操作

**操作安全（用于 Agent）：**
- 飞行前：1-2 行摘要意图操作和参数
- 执行：通过工具
- 飞行后：确认结果和执行的验证

### 3.8 研究三阶段模式

```
1. Plan：列出 3-6 个子问题
2. Retrieve：逐个搜索 + 追踪 1-2 个二阶线索（second-order leads）
3. Synthesize：解决矛盾 + 带引用写出答案
停止条件：更多搜索不太可能改变结论
```

"二阶线索"是关键概念——从初始搜索结果中发现新的搜索方向，实现**递归深入**而非浅层扫描。

### 3.9 引用与事实锚定

**引用规则：**
- 只引用当前工作流中检索到的来源
- 永不编造引用、URL、ID 或引用片段
- 引用附着到它支持的具体声明，而不仅仅放在末尾

**锚定规则：**
- 声明只基于提供的上下文或工具输出
- 来源冲突 → 显式说明冲突并归因各方
- 上下文不足 → 缩小答案范围或声明无法支持
- 推断而非直接支持的事实 → 标注为推断

## 四、专项工作流

### 4.1 编码与终端任务

**自主性与持久性：**
- 在当前轮次中端到端持续直到任务完全处理
- 不停在分析或部分修复，贯穿实现、验证、结果解释
- 除非用户明确要求计划/讨论，否则直接实现变更
- 遇到挑战/阻塞 → 自行尝试解决

**终端工具卫生：**
- 只通过终端工具运行 shell 命令
- 不将工具名当 shell 命令"运行"
- 有专用编辑工具就用它，不在 bash 中尝试
- 变更后运行轻量验证（ls/测试/构建）

### 4.2 前端设计任务

一组非常具体的 UI/UX 硬规则：

- **一个视口 = 一个构图**，不是仪表板（除非它就是仪表板）
- **品牌优先**：品牌名必须是 hero 级信号，不仅是导航文字
- **品牌测试**：去掉导航后，首屏能否被认出是哪个品牌？
- **全出血 Hero**：着陆页 hero 图应是主导的边到边视觉平面
- **Hero 预算**：首屏只放品牌+标题+一句话+CTA组+一张主图
- **不在 Hero 上覆盖浮动元素**
- **默认不用卡片**：如果去掉边框/阴影不影响理解，就不应该是卡片
- **每个区段一个职责**：一个目的、一个标题、一句说明
- **减少杂乱**：避免标签集群、统计条、图标行、盒装促销
- **有意义的动效**：2-3 个有意图的动效，优先用 Framer Motion

这说明 OpenAI 正在将**领域专家知识**编码到 prompt 模板中。

### 4.3 结构化输出

- 只输出要求的格式（SQL/JSON/XML）
- 不加散文或 markdown 围栏（除非被要求）
- 验证括号平衡
- 不编造表或字段
- 缺少 schema 信息 → 询问或返回显式错误对象

### 4.4 人格与写作控制

六维控制模型：
- Persona（一句话人设）
- Channel（Slack / email / memo / PRD / blog）
- Emotional register（情感基调 + "不要过度X"）
- Formatting（禁止列表/标题/markdown 等）
- Length（硬性限制）
- Default follow-through（低风险直接执行）

### 4.5 备忘录模式

- 精炼、专业的备忘录风格
- 使用精确名称、日期、实体、权威机构
- 精确结论优于笼统对冲
- 不确定时 → 关联到具体缺失事实或冲突来源
- 跨文档综合而非逐个总结

## 五、API 工程要点

### 5.1 Phase 参数

Responses API 的 `phase` 字段防止长任务过早停止：

- 可选但强烈推荐
- 用于长运行或工具密集的 agent
- 重放时保留 phase
- 不要给 user messages 加 phase
- 缺少 phase → 前言可能被误解为最终答案

### 5.2 Compaction（压缩）机制

- 在主要里程碑后压缩
- 压缩项视为不透明状态
- 压缩后保持 prompt 功能等价
- 端点返回 `encrypted_content` 用于后续请求
- 显著扩展有效上下文窗口

## 七、架构层面启示

### 7.1 Prompt 即配置（Prompt as Configuration）

模块化 prompt 架构：

```
系统 Prompt = 基础行为合约
            + 工具使用规则
            + 领域特定模块（研究/编码/前端/写作）
            + 个性化控制
            + 推理等级配置
```

每个模块是独立的 XML 块，可按需组合。

### 7.2 与行业实践的对比

| 维度 | 传统 Prompt Engineering | GPT-5.4 Prompt Guidance |
|------|----------------------|------------------------|
| 粒度 | 自然语言描述 | XML 合约块 |
| 工具使用 | "可以使用工具" | 持久性规则 + 依赖检查 + 重试策略 |
| 完成标准 | 隐含的 | `<completeness_contract>` 显式定义 |
| 安全性 | "请小心" | `<action_safety>` 飞行前/后检查 |
| 推理控制 | 无 | 五级推理旋钮 + "prompt first" 策略 |
| 输出控制 | "简洁一些" | `<verbosity_controls>` + `<output_contract>` |

## 八、核心 Takeaway

1. **Prompt 正在从"提示"演化为"系统规范"** — 合约化、模块化、可组合
2. **可靠性来自显式约束** — 模型越强大，越需要精确的行为边界
3. **推理算力是最后手段** — 结构化 prompt 的 ROI 远高于提高推理等级
4. **防御性设计是必需的** — 空结果重试、依赖前置检查、验证循环不是可选项
5. **领域知识可以编码为 prompt 模块** — 前端设计规则就是一个范例
6. **Agent 系统需要"阶段感知"** — phase 参数和 compaction 机制解决长程 agent 的核心挑战

这篇指南本质上是一份**"如何让 LLM 像可靠的工程系统一样运行"的操作手册**，对任何构建 AI agent 或复杂 LLM 应用的团队都有直接参考价值。

## 九、对 Baton 项目的启发

基于对 baton 项目的深入阅读（workflow.md、workflow-full.md、first-principles.md、research-agentic-rewrite.md、README.md），以下是 GPT-5.4 Prompt Guidance 对 baton 的具体启发：

### 9.1 已验证：Baton 做对了什么

GPT-5.4 指南从工业实践角度**印证了 baton 的多个核心设计决策**：

| Baton 设计 | GPT-5.4 对应模式 | 启示 |
|-----------|-----------------|------|
| Write Lock（BATON:GO） | `<action_safety>` 飞行前/后检查 | baton 的写锁本质上是 GPT-5.4 所说的"高影响操作的执行前验证"。这不是过度控制，而是工业级 agent 系统的标准做法 |
| Evidence Standards（file:line） | `<verification_loop>` + `<grounding_rules>` | GPT-5.4 要求"事实声明必须有上下文或工具输出支撑"，与 baton 的"every claim requires file:line"完全一致 |
| Phase Guidance（phase-guide.sh） | API `phase` 参数 | GPT-5.4 明确警告"缺少 phase 会导致前言被误解为最终答案"。baton 的阶段注入是正确的——agent 需要知道自己在哪个阶段 |
| Annotation Protocol（6 种标注） | `<completeness_contract>` + `<missing_context_gating>` | baton 的标注类型本质上是一套**结构化的缺失信息门控**，与 GPT-5.4 的"不猜，查或问"理念一致 |
| "Disagree with evidence" | `<default_follow_through_policy>` | GPT-5.4 也区分了"可直接执行"和"需要询问"的操作，但 baton 做得更彻底——要求 AI 在发现问题时主动用证据反驳 |
| 分层注入（workflow.md 400 tokens + phase-guide 动态注入） | 模块化 XML 合约块按需组合 | GPT-5.4 的模块化架构与 baton 的 Layer 0/1/2 设计异曲同工 |
| Session Handoff（Lessons Learned） | Compaction 机制 | 两者都在解决长程 agent 的状态保持问题，但方式不同：GPT-5.4 用加密压缩，baton 用人类可读的经验文档 |

### 9.2 可借鉴：Baton 可以吸收什么

#### A. 空结果恢复策略（`<empty_result_recovery>`）

**现状**：baton 的研究阶段要求"inventory all available documentation retrieval tools"（workflow-full.md:122），但没有规定工具返回空结果时该怎么办。

**建议**：在 RESEARCH 阶段引导中增加类似 GPT-5.4 的恢复策略：

```
如果工具搜索返回空结果或可疑的窄结果：
- 不要立即断定"不存在"
- 至少尝试 1-2 种备选策略（换关键词、放宽过滤、换工具）
- 然后才报告"未找到"+ 说明尝试了什么
```

这直接解决了 research-agentic-rewrite.md 中提到的"工具清点为研究第 0 步——已两次遗漏"的问题。

#### B. 依赖前置检查（`<dependency_checks>`）

**现状**：baton 的实现阶段有"dependency ordering"（workflow-full.md:347-349），但只描述了执行顺序，没有要求**执行前检查前置条件是否满足**。

**建议**：在 IMPLEMENT 阶段的 Self-Check Triggers 中增加：

```
执行每个 todo item 前：检查它依赖的前置 item 是否真的完成了，
以及前置 item 的实际输出是否与计划一致。
不要因为前置 item 标记了 [x] 就假设一切正确。
```

#### C. 完成度合约强化（`<completeness_contract>`）

**现状**：baton 有 completion-check.sh（hooks），但 workflow 文本中没有显式的完成度定义。

**建议**：在 IMPLEMENT 阶段增加类似 GPT-5.4 的完成度合约：

```
任务在以下条件全部满足前视为未完成：
- 所有 todo items 标记 [x]
- 每个 [x] item 经过自检（re-read code vs plan intent）
- 整体验证通过（typecheck/test）
- 被阻塞的 items 标记 [blocked] 并说明原因
```

#### D. 研究三阶段模式

**现状**：baton 的研究阶段有"observe-then-decide"策略（workflow-full.md:126），但没有明确的**多轮扫描**结构。

**GPT-5.4 模式**：Plan（列子问题）→ Retrieve（逐个搜索+追二阶线索）→ Synthesize（解决矛盾+写结论）

**建议**：作为策略提示（非强制），在 RESEARCH 阶段的 Strategy Hints 中增加：

```
对于复杂研究任务，考虑三轮扫描：
1. 规划：列出需要回答的 3-6 个子问题
2. 检索：逐个调查 + 追踪 1-2 个二阶线索（从初始发现中涌现的新方向）
3. 综合：解决矛盾，写出带证据的结论
```

这与 baton 的 agentic 改写方向一致——给启发式而非 prescriptive 步骤。

#### E. 进度更新规范（`<user_updates_spec>`）

**现状**：baton 没有定义 AI 在实现阶段应如何向人类报告进度。

**建议**：在 IMPLEMENT 阶段增加轻量级的进度更新指引：

```
重大阶段转换时或计划变更时更新人类。
每次更新：1 句结果 + 1 句下一步。
不要叙述常规工具调用。
```

### 9.3 需谨慎：不应直接照搬的

| GPT-5.4 模式 | 为什么不适合 Baton |
|-------------|-------------------|
| `<instruction_priority>` 指令优先级层级 | Baton 的设计是**人始终有最终决定权**，不需要定义"新指令覆盖旧指令"的自动规则——标注协议已经处理了冲突 |
| `<default_follow_through_policy>` 自动执行低风险操作 | 与 baton 的核心哲学冲突——baton 要求**先建立共同理解再行动**，不鼓励"看起来低风险就直接做" |
| `<parallel_tool_calling>` 并行工具调用 | Baton 已经有依赖排序规则（workflow-full.md:347-349），且 first-principles.md:427-452 深入讨论了 subagent 依赖问题。不需要额外的并行规则 |
| 前端设计规则 | 这是领域特定内容。Baton 是通用协议，不应包含特定领域的 prompt 模块 |
| Compaction（加密压缩） | 这是 API 层功能，baton 作为 prompt 层协议无法控制 |

### 9.4 架构层面的深层启示

#### "Prompt 即系统规范" 对 Baton 的意义

GPT-5.4 指南最重要的信号是：**工业级 AI 应用正在将 prompt 当作系统规范来设计**——合约化、模块化、可验证。

Baton 的 workflow.md 本质上已经是一个"系统规范"，但 research-agentic-rewrite.md 的分析显示它有 65% 的规则是 prescriptive 风格。GPT-5.4 指南提供了一个对比视角：

- GPT-5.4 的合约块是**声明式的**（定义"什么条件满足算完成"），不是过程式的（"先做X，再做Y"）
- Baton 的 agentic 改写方向（从 prescriptive → goal-driven + constraint）与 GPT-5.4 的合约式设计**高度一致**

这进一步验证了 research-agentic-rewrite.md 的核心结论：baton 应该更多使用目标驱动+约束式的规则，保留命令式仅用于硬约束（write-lock、BATON:GO、格式要求）。

#### "推理算力是最后手段" 对 Baton 的意义

GPT-5.4 指南的优先级排序（结构化 prompt > 输出合约 > 验证循环 > 提高推理等级）直接支持 baton 的设计理念：

> **Baton 不追求让 AI "更聪明"，而是通过结构（文档+标注+写锁）确保 AI 的智能被正确引导。**

这与 first-principles.md 的核心洞察完全一致："根本问题不是 AI 不够好，而是双方的信息没有充分交换。"
