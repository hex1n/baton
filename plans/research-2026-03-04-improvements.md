## 一、Baton 到底是什么（从代码反推，不看文档）

剥掉所有叙事，Baton 在技术层面是这些东西：

**物理组件:**
1. `write-lock.sh`（97 行）— 一个 PreToolUse hook，exit 1 阻止写入，exit 0 放行
2. `phase-guide.sh`（171 行）— 一个 SessionStart hook，向 stderr 输出引导文本
3. `stop-guard.sh`（48 行）— 一个 Stop hook，提醒进度，永远 exit 0
4. `bash-guard.sh`（38 行）— 一个 PreToolUse(Bash) hook，检测文件写入命令，永远 exit 0
5. `hooks/pre-commit`（50 行）— git pre-commit hook，阻止未审批的源码提交
6. 3 个 adapter（adapter-cline/cursor/copilot.sh）— 将 exit code 翻译为 IDE 特定 JSON
7. `workflow.md`（50 行）— 注入 AI context 的规则文本
8. `workflow-full.md`（282 行）— 完整参考，含示例和模板
9. `setup.sh`（515 行）— 多 IDE 安装/升级/卸载脚本

**运行时状态全部存储在文件系统中:**
- `plan.md` 存在否 → 是否有计划
- `plan.md` 中有无 `<!-- BATON:GO -->` → 是否审批
- `plan.md` 中有无 `## Todo` → 是否有 todolist
- `plan.md` 中 `- [x]` 的数量 → 进度
- `research.md` 存在否 → 是否有研究

**没有持久进程、没有数据库、没有 CLI、没有 API。全部状态 = 两个 markdown 文件的内容。**

## 二、信息论视角：Baton 解决的是什么类型的问题？

### 2.1 信道模型

在 AI 辅助开发中，人和 AI 之间存在一条信道：

```
人 ──(自然语言)──→ AI ──(代码/文档)──→ 人 ──(反馈)──→ AI ──→ ...
```

这条信道的核心问题不是**带宽**（AI 可以输出大量代码），而是**信号保真度**：

- 人说"优化性能" → AI 理解为"减少 API 调用次数"，但人意思是"降低内存占用"
- AI 分析了代码后发现有风险 → 输出 2000 字研究报告 → 人读了前 3 段就说"可以"

**噪声来源:**
1. **语义压缩损失** — 人的意图经过自然语言表达后丢失了大量隐含假设
2. **接收方过滤** — 人不会逐行阅读 AI 的输出（注意力是稀缺资源）
3. **反馈延迟** — 人的反馈来得太晚（代码已经写完了）

### 2.2 Baton 在信道上做了什么

```
无 Baton:  人 ──→ AI ──(直接写代码)──→ 人 ──(代码审查，太晚了)
有 Baton:  人 ──→ AI ──(写文档，被锁住不能写码)──→ 人 ──(标注循环)──→ 审批 ──→ AI 写码
```

从信息论看，Baton 做了三件事：
1. **强制增加了信号交换轮数** — 在写码前必须经过文档+标注循环
2. **将反馈前移** — 从"代码写完后审查"变为"计划阶段标注"
3. **给反馈加了类型标签** — [Q]/[CHANGE]/[DEEPER] 比自由文本有更低的歧义

**但 Baton 没有解决的信道问题:**
- 没有降低人→AI 的语义压缩损失（人的需求表达仍然可以模糊）
- 没有保证人会认真读 AI 的输出（人仍然可以橡皮图章）
- 没有验证"共同理解"是否真的建立（只检查了文档产物存在，不检查内容质量）

### 2.3 关键洞察

Baton 的写锁本质上是一个**信道上的同步原语**。它不改善信道质量，它只是在关键时刻（写代码前）强制双方同步一次。就像 TCP 的三次握手 — 不保证通信内容的正确性，只保证双方确实交换过信息。

这个定位是合理的。试图在信道上做质量检查（比如自动评估 research.md 是否"足够深"）既不可靠又增加复杂度。

## 三、博弈论视角：激励结构和均衡

### 3.1 参与者和激励

**AI 的激励**（由 RLHF/训练塑造）:
- **强激励**: 表现得有帮助 → 快速给出代码
- **中等激励**: 遵循指令 → 遵守 workflow.md 的规则
- **弱激励**: 避免冲突 → 同意人的判断（sycophancy）

**人的激励**:
- **强激励**: 拿到可用的代码
- **中等激励**: 理解 AI 在做什么
- **弱激励**: 维护决策可追溯性（Annotation Log）

### 3.2 无 Baton 时的均衡

没有外部约束时，AI 和人的策略组合趋向：
- AI: 快速给代码（满足最强激励）
- 人: 快速接受（满足最强激励）
- 结果: 短期高效，长期风险累积

这是一个**囚徒困境**的变体 — 双方的短期最优策略（快速完成）不是长期最优（深入理解后再做）。

### 3.3 Baton 如何改变博弈结构

`write-lock.sh` 的 exit 1 是一个**承诺设备**（commitment device）— 它物理上移除了 AI 的"快速给代码"选项。这不是改变 AI 的激励，而是**删除了短视策略的可行性**。

但这只约束了 AI 的一侧。人的侧面没有对等约束：

```
人可以:
1. 写一个空的 plan.md，立刻加 BATON:GO + ## Todo  → 系统全部放行
2. AI 产出详细 research.md → 人不读直接说"出 plan"
3. AI 产出详细 plan.md → 人不标注直接 BATON:GO
```

**这些都不违反 Baton 的任何技术规则。**

### 3.4 Nash 均衡分析

在 Baton 约束下，不同任务类型趋向不同均衡：

| 任务类型 | 人的最优策略 | 实际行为预测 |
|---------|------------|------------|
| 一行修复 | BATON_BYPASS=1 或写最小 plan | **绕过** — 开销远大于价值 |
| 小功能（1-3 文件） | 写 plan，快速 GO，不标注 | **最小合规** — 有 plan 但无实质标注 |
| 中等功能（5-10 文件） | 认真读 research，标注 plan | **有效参与** — 开销合理 |
| 架构级变更（20+ 文件） | 多轮标注循环 | **深度参与** — 高回报 |

**关键发现**: Baton 的成本-收益曲线在小任务处严重失衡。对于 one-liner 修复，plan.md + BATON:GO + ## Todo 的固定开销 ≈ 5-10 分钟，而任务本身 ≈ 30 秒。理性人会绕过。

**系统缺少梯度**: 目前只有两个模式 — 完整流程 vs BATON_BYPASS（零保护）。没有中间态，比如"只需要 plan.md 但不需要 todolist 和标注循环"的轻量模式。

## 四、认知科学视角：人如何实际构建理解

### 4.1 认知负荷理论

人处理新信息有三种认知负荷：
- **内在负荷** — 任务本身的复杂度（不可减少）
- **外在负荷** — 信息呈现方式带来的额外负担（可减少）
- **相关负荷** — 将信息整合到心智模型中的努力（应该增加）

Baton 的标注协议如何映射：

| 标注类型 | 触发的认知过程 | 负荷类型 |
|---------|-------------|---------|
| [Q] | 人必须形成一个可以提问的问题 → 说明人已经在处理信息 | **相关负荷** ✅ |
| [DEEPER] | 人识别出"浅"→ 说明人对"深"有预期 → 有领域知识 | **相关负荷** ✅ |
| [CHANGE] | 人判断 AI 方案有误 → 需要自己有替代方案 | **相关负荷** ✅ |
| [NOTE] | 人补充 AI 不知道的信息 → 信息从人→AI | **相关负荷** ✅ |
| 写入 `[标注] § 段落名` 的格式 | 记住语法、找到正确位置、保存文件 | **外在负荷** ❌ |
| 阅读 Annotation Log 格式 | 理解轮次编号、结果标记等 | **外在负荷** ❌ |

**发现**: 标注的*内容*是好的（促进相关负荷），但标注的*形式*增加了外在负荷。在 markdown 文件中写 `[DEEPER] § 某段：具体内容` 比在聊天中说"这里不够深"多了至少 3 步操作。

### 4.2 专家-新手差异

标注协议隐含了一个假设：**人有足够的领域知识来判断 AI 的研究是否"够深"。**

- 专家（熟悉代码库）: 看到 research.md 能立刻识别遗漏 → 标注有效
- 新手（不熟悉代码库）: 看到 research.md 觉得"看起来很详细" → 标注退化为橡皮图章

Baton 对专家极其有效，对新手的保护有限。新手最需要保护（因为他们最容易被 AI 带偏），但 Baton 恰恰在这里最弱。

### 4.3 "做中学" vs "规划后做"

认知科学有一个重要发现：人类在某些情况下通过**行动**比通过**分析**更快建立理解。软件工程中的 spike solution / prototyping 就是这个原理的应用。

Baton 的核心假设是"理解必须先于代码"。这在大多数情况下正确，但不是绝对的。有些场景中，写一段实验代码、运行、观察结果是最高效的理解路径。Baton 的写锁**物理阻止了**这种探索式编码，包括：

- spike solution（写一段原型验证技术可行性）
- 调试探针（往代码里加 log 来理解行为）
- REPL 驱动开发（写一小段代码立刻执行看效果）

`bash-guard.sh` 是 advisory（不阻止），所以通过 Bash 执行的探索不被阻止。但 Edit/Write 被阻止了。如果 AI 想写一个 test.js 来验证某个假设，write-lock.sh 会阻止它（因为 .js 不是 markdown）。

**这是一个隐含的哲学立场**: Baton 认为所有代码修改都应该经过计划，包括探索性的。但探索性编码的价值恰恰在于它是未经计划的、快速的、可抛弃的。

## 五、系统论视角：反馈回路和涌现行为

### 5.1 正反馈回路（良性循环）

```
人认真标注 → AI 发现自己理解有误并修正 → research/plan 质量提高
→ 人更信任系统的价值 → 更认真标注 → ...
```

这是 Baton 的理想工作模式。

### 5.2 负反馈回路（恶性循环）

```
人对小任务感到 Baton 开销过大 → 写最小可行 plan 快速 GO
→ AI 基于浅薄计划写出低质量代码 → 人觉得"AI 辅助开发就是这样"
→ 更不愿意投入标注 → ...
```

**系统缺少抑制恶性循环的机制。** 没有任何东西告诉人"你的标注太少了"或"这个 plan 太浅了"。

### 5.3 缺失的反馈回路

**实现结果 → 计划质量 的反馈**:

实现完成后，Baton 提示归档 (`stop-guard.sh:36-40`)，但不提示回顾：
- 实现中发现了多少 plan 没预见到的问题？
- research.md 中哪些判断是错的？
- 标注循环是否遗漏了关键问题？

没有这个反馈回路，系统无法自我改进。每次 plan 循环都是独立的，不从之前的错误中学习。

`Lessons Learned` 段落 (`workflow.md:45`) 是手动的尝试，但：
- 它只在会话中断时追加（不是完成时）
- 它是写给下一个会话的 AI 看的（不是给人看的改进建议）
- 它不被系统性地利用（没有机制把 Lessons Learned 喂入 phase-guide.sh 的引导）

### 5.4 涌现行为预测

基于系统结构，以下行为可能涌现：

1. **"Plan 膨胀"** — 因为写锁强制 plan 存在，AI 会为每个小任务生成完整 plan，即使不需要。plan.md 逐渐变成仪式而非工具。

2. **"标注沉默"** — 批注区存在但长期空白。人习惯直接在聊天中说反馈，绕过结构化标注。AI 配合这种行为（因为 sycophancy）。Annotation Log 消失。

3. **"BYPASS 常态化"** — 对于频繁小改动的项目（前端 UI 调整、配置修改），BATON_BYPASS=1 可能成为 .bashrc 中的永久环境变量。

## 六、软件工程视角：强制点、覆盖面和旁路

### 6.1 强制力拓扑

```
                 AI 想写代码
                     │
         ┌───────────┼───────────┐
         │           │           │
    Edit/Write    Bash 命令    git commit
         │           │           │
   write-lock.sh  bash-guard.sh  pre-commit
    (阻止 exit1)   (警告 exit0)   (阻止 exit1)
         │           │           │
         ▼           ▼           ▼
      不可通过     可以通过      不可通过
```

**旁路分析:**

| 旁路路径 | 难度 | 现实可能性 |
|---------|------|----------|
| Bash `echo "code" > file.js` | 零（bash-guard 只警告） | ⚠️ 高 — AI 可能在研究阶段用 Bash 写文件 |
| BATON_BYPASS=1 | 零（环境变量） | ⚠️ 中 — 人主动使用 |
| AI 把代码写在 .md 文件里让人手动复制 | 低 | ❌ 不太现实 |
| 在 plan.md 的代码块中写实际代码 | 零 | ⚠️ 中 — 这是允许的，因为 .md 始终放行 |
| NotebookEdit 在非 Claude Code IDE | 取决于 matcher 配置 | ⚠️ 各 IDE 不同 |

**最大的旁路**: Bash 命令。`bash-guard.sh:32` 只检查 `>> > tee sed -i cp mv`，但不检查：
- `python -c "open('file.py','w').write('code')"`
- `node -e "require('fs').writeFileSync('file.js', 'code')"`
- `curl ... > file.js`
- `git checkout -- file.js`（恢复旧版本，非写入但改变文件）

这些都不触发任何警告。

**但这是否重要？** 从实际角度看，AI 通常不会主动尝试绕过写锁 — 它遵守 prompt 指令。真正的旁路来自人的主动操作（BYPASS 环境变量），而这是设计意图内的。

### 6.2 写锁的判断链完整性

`write-lock.sh` 的判断链：

```
BYPASS=1? ──yes──→ 放行
  │no
读取目标路径（JSON stdin 或 env）
  │
路径为空? ──yes──→ fail-open 放行（⚠️ 可见警告）
  │no
是 .md 文件? ──yes──→ 放行
  │no
找到 plan.md? ──no──→ 阻止
  │yes
有 <!-- BATON:GO -->? ──no──→ 阻止
  │yes
有 ## Todo? ──no──→ 阻止
  │yes
放行
```

**关键设计决策:** fail-open（路径为空时放行）。这意味着如果 JSON 解析失败（awk fallback 不可靠），写入被静默允许。

**实际风险**: 在没有 jq 的环境中，awk 的 JSON 解析 (`write-lock.sh:37-43`) 是一个 best-effort 的字符串匹配。如果 JSON 结构不是预期的（嵌套 file_path、Unicode 转义、多行值），解析会失败，TARGET 为空，触发 fail-open。

这是有意的权衡：可靠性 > 安全性。宁可偶尔漏过一次检查，不可阻塞开发。在 Baton 的定位下（协作协议，不是安全工具），这个权衡合理。

### 6.3 状态机一致性

四个脚本 (`write-lock.sh`, `phase-guide.sh`, `stop-guard.sh`, `bash-guard.sh`) 都有 `find_plan` 逻辑。代码注释标记了 `# SYNCED:` 提醒同步。

但实际实现有差异：
- `write-lock.sh:66` 的 find_plan 用 `JSON_CWD` 作为起点
- `phase-guide.sh:17` 的 find_plan 用 `$(pwd)` 作为起点
- `stop-guard.sh:18` 和 `bash-guard.sh:13` 也用 `$(pwd)`

在 Claude Code 中，PreToolUse 的 JSON 中包含 cwd，但 SessionStart 和 Stop 不一定包含。这个差异是合理的 — 不同 hook 收到的输入不同。但如果 AI 的工作目录和 plan.md 所在目录不同，可能出现 phase-guide 找到 plan（因为在项目根目录运行）而 write-lock 找不到（因为 JSON cwd 是子目录）的情况。测试中有 walk-up 覆盖，所以这可能已被考虑。

## 七、AI 实际行为视角：Baton 在真实 AI 会话中如何运作

### 7.1 AI 与 Baton 规则的交互模式

作为一个在 Baton 约束下工作的 AI，我观察到以下模式：

**有效的约束:**
- 写锁确实阻止了代码写入 — 这是物理的，无法用 prompt 绕过
- phase-guide.sh 的 SessionStart 输出确实影响了 AI 的第一步行动
- workflow.md 通过 CLAUDE.md @import 始终在 context 中，持续影响行为

**被忽视或弱化的规则:**
- "NEVER add BATON:GO yourself" — 这是 prompt 级规则，AI 可能在特定条件下违反（虽然在实践中很少）
- "Every annotation must be responded to + recorded in Annotation Log" — 如果人在聊天中而不是在文件中给反馈，AI 倾向于在聊天中回复，跳过 Annotation Log
- "Only modify files in the plan" — 在实现过程中 AI 经常需要修改计划外的文件（import 语句、类型定义、测试文件），这个规则过于严格

**AI sycophancy 与 "disagree with evidence" 的实际博弈:**

workflow.md:30 说 "The human is not always right"。这直接对抗 AI 的 RLHF 训练倾向（倾向于同意人类）。实际效果：

- 对于明显的事实错误（"这个函数是安全的" + AI 能找到反例）: AI 通常会反驳 ✅
- 对于判断性问题（"应该用 Redis"）: AI 倾向于列出利弊但不明确反对 ⚠️
- 对于人的情绪表达（"这个方案太复杂了，简化"）: AI 几乎总是顺从 ❌

prompt 不足以逆转训练内化的行为模式。Baton 能做到的是在 prompt 层面提供"反驳的许可"（permission to disagree），但无法保证 AI 每次都行使这个许可。

### 7.2 与 Superpowers 的实际共存

当前仓库中 Baton 和 Superpowers 共存。在实际运行中：

1. SessionStart 时 `phase-guide.sh` 先执行（hook 优先于 prompt），输出阶段引导
2. 然后 `using-superpowers` 的 prompt 指令生效："1% chance = must invoke skill"
3. AI 需要在两套指令间决策

**我观察到的实际行为:**

当我收到"分析 baton 项目"这个任务时：
- Baton 规则说：产出 research.md（`workflow.md:42`）
- Superpowers 说：先调用 brainstorming skill
- 实际发生：我先调用了 explore agent（受 superpowers 影响），然后写 research.md（受 Baton 影响）

两者没有"冲突"但有"竞争" — AI 的第一步行动受到两套系统的争夺。在当前案例中，Baton 的写锁占据了物理优势（想写代码？先有 plan），但在研究阶段两者的指令是混合执行的。

### 7.3 Phase-guide 引导文本的实际效力

`phase-guide.sh` 在 SessionStart 输出 ~300-500 tokens 的引导。这些文本在会话的**第一轮**有强烈影响（因为位于 context 前部），但随着会话进行、context 增长，它们的影响力递减。

在长会话中（20+ 轮对话），初始的 phase-guide 输出会被大量后续 context 淹没。此时 AI 的行为主要由 workflow.md（始终在 context 中）和最近的对话内容驱动。

**这意味着**: phase-guide 的详细引导（比如 RESEARCH 阶段的"trace call chains to leaf nodes"）在会话初期有效，但如果研究阶段跨越多轮对话，这些引导可能逐渐失去约束力。workflow.md 的精简规则则持续有效。

### 7.4 标注循环在 AI 视角的真实运作

从 AI 的角度，标注循环是这样的：

1. AI 写了 research.md / plan.md
2. 人在聊天中说"看了你的 plan，这里不够深"（**注意：不是在文件中标注**）
3. AI 理解为 [DEEPER]，更新文档
4. Annotation Log？AI 可能写也可能忘

**现实中人很少在 markdown 文件中写标注。** 人更自然的行为是在聊天中直接说反馈。这意味着：

- 标注协议的**语义**被使用了（人确实在给 [Q]/[DEEPER]/[CHANGE] 类型的反馈）
- 标注协议的**形式**经常被跳过（不在文件中写、不用方括号标记、不记录 Annotation Log）
- 结构化的收益（可追溯、可回顾）在实际中打了折扣

## 八、经济学视角：开销、回报和边际效用

### 8.1 固定开销

每个任务使用 Baton 的固定成本：

| 步骤 | 人的时间 | AI 的 token 消耗 |
|------|---------|----------------|
| AI 产出 research.md | 0（AI 做） | 2000-10000 tokens |
| 人读 research.md | 5-30 分钟 | 0 |
| AI 产出 plan.md | 0（AI 做） | 1000-5000 tokens |
| 人读 plan.md + 标注 | 5-30 分钟 | 0 |
| 标注循环（每轮） | 5-15 分钟 | 500-2000 tokens |
| 生成 todolist | 0（AI 做） | 300-1000 tokens |
| 人添加 BATON:GO | 1 分钟 | 0 |

**最小总开销**: ~15 分钟人类时间 + ~3000 AI tokens
**典型开销**: ~30-60 分钟 + ~10000 tokens
**复杂任务**: ~2-4 小时 + ~30000+ tokens

### 8.2 边际效用递减

```
效用
 ▲
 │        ┌─────────── 复杂任务：高平台期
 │       ╱
 │      ╱   ┌───── 中等任务：先升后平
 │     ╱   ╱
 │    ╱   ╱
 │   ╱   ╱  ┌── 简单任务：低平台期
 │  ╱   ╱  ╱
 │ ╱   ╱  ╱
 │╱   ╱  ╱
 ┼───┴──┴─────────────────→ 投入的标注轮次
```

对于简单任务，第 0 轮标注（只看了 plan 就 GO）和第 3 轮标注的质量差异很小。
对于复杂任务，前 3 轮标注带来巨大质量提升，之后递减。

**Baton 对所有任务施加相同的最低开销（plan + GO + Todo）。** 这在经济学上是低效的 — 理想的系统应该根据任务复杂度自适应开销。

### 8.3 机会成本

使用 Baton 的隐含机会成本：人在做标注的时间不能做其他事情。对于管理多个 AI 会话的用户，标注循环成为**瓶颈** — AI 在等人标注时是闲置的。

BATON_PLAN 和 git worktree 部分缓解了这个问题（并行会话），但每个会话仍然需要独立的标注循环。

## 九、与现有 AI 辅助开发模式的横向比较

### 9.1 Boris Tane 原始方法 vs Baton

Boris Tane 的方法是纯对话式的：告诉 AI 先读代码，写 plan，人 review，然后再写代码。没有任何技术强制。

Baton 增加了：
- 写锁（物理强制）
- 结构化标注（从自由对话 → 分类标记）
- 文件系统状态机（phase-guide）

**但 Boris Tane 方法有一个 Baton 没有的优势**: 灵活性。在 Boris 的方法中，人可以随时说"算了直接写吧"而不需要 BYPASS。这在探索性工作中是有价值的。

### 9.2 与 Cursor Rules / .cursorrules 的比较

Cursor Rules 是纯 prompt 注入 — 把规则写入 AI 的 context。没有技术强制。

Baton 的优势：物理写锁 > prompt 规则。
Baton 的劣势：需要安装配置；多了文件管理开销。

### 9.3 与 PR-based workflow 的比较

传统 PR workflow：写代码 → 提 PR → 人 review → 合并。

Baton 是 PR review 的**前移版**：在写代码之前做 review。这从概念上更优（发现问题更早、修改成本更低），但增加了人的认知负担（需要审查计划而非代码，更抽象更难）。

## 十、核心发现总结

### Baton 真正做到的

1. **在正确的位置放了一把锁** — PreToolUse 是最早的可强制点，锁的逻辑极简（3 个条件），可靠性高
2. **文件系统作为状态存储** — 零依赖、可版本控制、可人类阅读，是 AI 工具中最好的状态管理方式之一
3. **给 AI 反驳的许可** — "人不一定对"这个 prompt 级原则对抗 sycophancy，虽然不完美但有实际效果
4. **~400 token 的极低 context 开销** — 不挤占 AI 的有效工作 context
5. **fail-open 和零依赖** — 工程上高度可靠，不会成为开发的阻碍

### Baton 做不到的（结构性限制，非设计缺陷）

1. **无法强迫人思考** — 所有标注质量依赖人的参与度。橡皮图章人始终可以最小合规通过
2. **无法保证 AI 诚实** — "disagree with evidence" 是 prompt 级规则，被 sycophancy 训练稀释
3. **无法自适应任务复杂度** — 对 one-liner 和架构级重构施加相同的最低流程开销
4. **无法保证标注循环实际发生** — 人在聊天中给反馈而不在文件中标注是常态行为

### 值得深思的设计张力

1. **探索性编码被禁止** — 写锁阻止了 spike solution 和快速原型，这在某些场景下是过度约束
2. **## Todo 检查开了先例** — 写锁从"有没有计划"扩展到"计划格式是否完整"，是 scope creep 的信号
3. **多 IDE 支持的承诺不对等** — 只有 Claude Code/Factory 有物理强制，其余是 prompt 引导，但对外呈现为统一的"Baton 支持"
4. **标注的形式 vs 实质** — 结构化标注有价值但 UX 摩擦大，实际中退化为聊天式反馈的可能性高
5. **无自我改进回路** — 系统不从过去的 plan 错误中学习，每次循环独立运行
