# Plan: Baton HOTL 终极版 — 基于复杂度的分级审批架构

**Complexity**: Large (10+ files, architectural change)
**Research**: research.md §3 HOTL 转型路径 + §6 HITL 审计

---

## 设计理念

将 baton 从 **纯 HITL**（每次写代码都需 BATON:GO）演进为 **HITL + HOTL 混合模式**：

- **Trivial/Small** 任务：AI 高置信度时自动放行，人类事后审阅
- **Medium/Large** 任务：保持现有 BATON:GO 审批流程不变
- 人类可随时通过 `BATON_REGULATORY=1` 禁用自动审批（强合规场景）
- 人类可在任何自动审批后追溯覆盖

**核心不变**：plan-first workflow、结构化反馈协议、evidence standard、复杂度校准概念。

## 变更清单

### 变更 1: write-lock.sh v4 — 分级审批门控

**文件**: `.baton/hooks/write-lock.sh`
**当前**: v3.0, 99 行。逻辑：fail-open trap (L14) → BATON_BYPASS (L17) → JSON stdin 解析 (L22-44) → markdown 豁免 (L56-58) → find_plan 上溯 (L71-79) → BATON:GO grep (L91-93) → 阻止 (L96-98)
**变更**: 在 L93 (BATON:GO exit 0) 之后、L96 (阻止) 之前插入 HOTL 分级审批逻辑

**逻辑流程**：
```sh
# 阶段 0: 现有逻辑不变
BATON_BYPASS=1 → exit 0
*.md → exit 0
BATON:GO in plan → exit 0

# 阶段 1: 合规模式（新增）
BATON_REGULATORY=1 → 必须 BATON:GO，跳过自动审批

# 阶段 2: 解析 plan 中的 HOTL 元数据（新增）
从 plan.md 中读取:
  <!-- HOTL: complexity=TRIVIAL confidence=0.96 -->
如果格式不匹配或缺失 → 走旧逻辑（需要 BATON:GO）

# 阶段 3: 风险模式检测（新增）
检查目标文件的扩展名和路径:
  *.sql, *migration*, *secret*, *credential*, *.env → 强制 BATON:GO

# 阶段 4: 分级审批（新增）
TRIVIAL + confidence ≥ 0.95 → exit 0 + stderr 提示 "⚡ Auto-approved"
SMALL   + confidence ≥ 0.90 → exit 0 + stderr 提示 "⚡ Auto-approved"
其他 → exit 1 + 现有阻止消息
```

**HOTL 元数据格式**（写在 plan.md 中）：
```markdown
<!-- HOTL: complexity=TRIVIAL confidence=0.96 -->
```
- AI 在写 plan 时生成此标记
- 人类审阅 plan 时可见此标记，可修改或删除
- 删除此标记 = 强制走 BATON:GO 流程
- AI **不得自行添加 BATON:GO**，但可以添加 HOTL 元数据（因为这只是评估，不是审批）

**风险模式列表**（硬编码在 write-lock.sh 中）：
```
*.sql *.env .env* *secret* *credential* *password* *migration*
*docker-compose* *Dockerfile* *.yml (CI/CD) *.yaml (CI/CD)
```
匹配这些模式的文件 → 无论复杂度和置信度，必须 BATON:GO。

### 变更 2: audit-log.sh — 审计日志系统（新增）

**文件**: `.baton/hooks/audit-log.sh`（新文件）
**用途**: 被 write-lock.sh 和 post-write-tracker.sh 调用，写入结构化审计记录

**日志位置**: `.baton/audit.jsonl`（JSON Lines 格式，每行一条记录）

**记录格式**：
```json
{
  "ts": "2026-03-07T14:32:00+08:00",
  "file": "src/cache.ts",
  "plan": "plan.md",
  "complexity": "TRIVIAL",
  "confidence": 0.96,
  "approval": "auto",
  "risk_flags": [],
  "tool": "Edit"
}
```

`approval` 字段值:
- `"auto"` — HOTL 自动放行
- `"manual"` — BATON:GO 人类审批
- `"bypass"` — BATON_BYPASS=1

**实现**：纯 shell，一个 `log_audit()` 函数，被其他 hook 调用或 source。

**Git 忽略**: `.baton/audit.jsonl` 加入 `.gitignore`（审计日志是本地的，不提交）。

## 风险分析

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| AI 将 Medium 任务错标为 Trivial | 🔴 高 | HOTL 元数据在 plan.md 中可见，人类可修改；风险文件强制 BATON:GO |
| 自动审批的代码有 bug | 🟡 中 | 审计日志 + stop-guard 汇总提醒；人类可追溯审阅 |
| HOTL 元数据被滥用 | 🟡 中 | BATON_REGULATORY=1 全局禁用；风险模式硬编码 |
| 审计日志膨胀 | 🟢 低 | JSONL 格式，每条 ~200 字节；.gitignore 排除 |
| OpenCode adapter 与 write-lock.sh 不一致 | 🟡 中 | test-workflow-consistency.sh 新增一致性检查 |
| Skill 文件与 phase-guide fallback HOTL 文本不同步 | 🟡 中 | HOTL 指导分布在 3 处（2 skill + 1 fallback），需一致性测试覆盖 |

## Self-Review

- **最大风险**: 置信度是 AI 自评的——AI 可能系统性高估自己的置信度。缓解措施是风险文件硬编码阻止 + 人类可删除 HOTL 标记，但日常使用中人类可能逐渐信任而不检查。
- **可能完全错误的假设**: 假设 HOTL 元数据格式 `<!-- HOTL: complexity=X confidence=Y -->` 能可靠解析。如果 AI 写出变体格式（多空格、换行），shell 解析会失败 → 回退到 BATON:GO（fail-safe，不是 bug）。
- **被否决的替代方案**: 考虑过用独立的 `.baton/hotl.json` 配置文件存储 HOTL 元数据，但这会引入第二状态源（plan.md + hotl.json），违反 "single source of truth" 原则。元数据内嵌在 plan.md 中，人类审阅 plan 时自然可见。
- **会话边界方案**: 使用 `.baton/.hotl-session-start` 临时文件标记首次自动审批时间，stop-guard.sh 据此过滤 audit.jsonl。比 env var 可靠（子进程继承问题），比 PID 文件简单。缺点：如果 stop-guard 未运行（如 IDE 崩溃），临时文件残留 → 下次会话汇总可能包含旧数据。可接受（fail-noisy，不是 fail-silent）。
