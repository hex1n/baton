# Baton 第一性原理分析

> 日期: 2026-03-02
> 方法: 不从现有方案出发，从根本问题出发推导设计

---

## 一、根本问题是什么

AI 辅助开发中，有一个结构性矛盾：

```
人类知道：意图、业务上下文、优先级、历史坑
AI 知道：代码结构、调用链、模式、实现细节

双方都只有一半的信息。
```

当 AI 基于不完整的信息写代码时，结果可能：
- 技术上可运行，但方向错误（不是人想要的）
- 表面正确，但忽略了深层约束（人知道的坑）
- 改了一个地方，但没意识到对其他地方的影响（研究不够深）

当人类没有充分理解 AI 的分析时，审批可能：
- 橡皮图章（看不懂就通过）
- 错误干预（基于过时理解给出错误指导）

**根本问题不是"AI 不够好"或"人不够细心"，而是双方的信息没有充分交换。**

---

## 二、解决这个问题需要什么

从根本问题出发，解决方案必须实现一件事：

> **在 AI 动手写代码之前，人和 AI 必须建立对任务的共同理解。**

"共同理解"的含义：
- AI 理解了人的意图（不只是字面需求，还包括背后的原因和约束）
- 人理解了 AI 的分析（代码现状、改动方案、风险、取舍）
- 双方对"该做什么、为什么这么做、有什么风险"达成一致

**如果共同理解没有建立，任何代码输出都是赌博。**

---

## 三、共同理解怎么建立

人和 AI 建立共同理解，只有一种方式：**对话。**

但普通对话（聊天）有四个致命缺陷：

| 缺陷 | 表现 |
|------|------|
| **短暂** | 早期讨论的关键决策滑出 context window，AI 忘记了 |
| **无结构** | 哪些是已确认的、哪些是待讨论的、哪些是被否定的，混在一起 |
| **无强制** | AI 可以在理解不充分时就开始写代码 |
| **不可回溯** | 事后无法追踪"为什么做了这个决定" |

所以我们需要的是：**持久化的、结构化的、有强制力的对话机制。**

这就是 Baton 要做的事。

---

## 四、推导 Baton 的最小必要机制

从"持久化的、结构化的、有强制力的对话机制"出发，逐一推导：

### 机制 1：持久化 → 文档

对话内容不能只存在于聊天记录中，必须沉淀为文档。

文档是"共同理解"的载体：
- **research.md**（或 `research-<topic>.md`）— AI 的理解（代码现状、调用链、风险、模式）
- **plan.md**（或 `plan-<topic>.md`）— 双方的共识（做什么、为什么、怎么做、风险应对）

为什么是两个文档而不是一个？因为它们的功能不同：
- research.md 是**AI 向人传递信息**（"这是我对代码的理解"）
- plan.md 是**双方达成共识**（"我们同意这么做"）

research 可能很长、很技术。plan 应该清晰、可判断。

### 机制 2：结构化 → 标注协议

对话需要结构，否则会退化成混乱的你一句我一句。

结构化的核心是：**每条反馈都有明确的类型和预期的回应方式。**

这就是标注（Annotation）：

```
人在文档中写一条标注 → AI 必须回应 → AI 的回应被记录 → 人决定是否满意
```

标注需要分类型，因为不同类型的反馈需要不同的回应：

| 类型 | 人的意图 | AI 应该做什么 |
|------|---------|-------------|
| 提问 | "我不理解这个" | 解释，用证据 |
| 补充 | "你漏了这个信息" | 纳入，更新文档 |
| 纠正 | "你理解错了" | 检查证据，承认或用证据反驳 |
| 要求改变 | "我想换个方向" | 评估影响，如果有问题要说出来 |
| 要求深入 | "这里不够深" | 继续调查，更新文档 |

### 机制 3：有强制力 → 写锁

对话必须在行动之前完成。否则 AI 会跳过对话直接写代码。

**写锁的本质不是"控制 AI"，而是"保证时序"**：共同理解必须先于代码修改。

写锁是最简单的实现：plan.md 存在且被审批 → 放行。否则 → 阻止。

### 机制 4：可回溯 → 标注日志

每轮对话（标注 + 回应）必须被记录，原因：
- 后续标注可能依赖前面的决策
- 实现阶段可以回溯"为什么这么设计"
- 如果 AI 的回应有误，可以追踪到何时何地出了问题

---

## 五、从机制推导出 Baton 的完整设计

### Baton 是什么

**Baton 是一个用于 AI 辅助开发的"共同理解构建协议"。**

它通过三个组件实现：
1. **文档**（research.md / plan.md）— 共同理解的载体
2. **标注协议**（annotation protocol）— 结构化对话的方式
3. **写锁**（write lock）— 确保共同理解先于代码修改

### Baton 不是什么

- 不是工作流引擎（不规定阶段顺序）
- 不是方法论（不规定怎么研究、怎么实现）
- 不是项目管理工具（不跟踪任务状态）
- 不是质量门控（不检查文档格式是否达标）

**Baton 只做一件事：确保人和 AI 在动手之前充分交换了信息。**

### 完整流程

不同场景走不同路径，但底层协议相同：

#### 场景 A：目标明确

```
人的意图已经清晰（"重构这个函数"、"新增XX功能"）

① AI 深度研究代码 → 产出 research.md
   · AI 向人传递：代码现状、调用链、风险、模式
   · 人阅读 research.md，获得对代码的理解

② 人提出需求

③ AI 基于 research + 需求 → 产出 plan.md
   · plan.md = AI 的方案 + 理由 + 取舍 + 风险

④ 人标注 plan.md ←→ AI 回应 → 循环
   · 共同理解在这里建立
   · 直到人满意

⑤ 人："生成 todolist" → AI 追加 ## Todo
⑥ 人："开始" / <!-- BATON:GO --> → 写锁解除 → 实现 → typecheck
```

#### 场景 B：需要探索

```
人的意图还不清晰（"我有个想法..."、"这里有 bug..."）

① 人描述想法/问题
② AI 调研 → 产出 research.md

③ 人标注 research.md ←→ AI 回应 → 循环
   · "这里不够深" → AI 继续挖
   · "你理解错了" → AI 修正
   · "还漏了XX" → AI 补充
   · 共同理解在这里逐步建立

④ 研究清晰后，人："出 plan"
⑤ AI 产出 plan.md

⑥ 人标注 plan.md ←→ AI 回应 → 循环

⑦ 生成 todolist → 实现 → typecheck
```

#### 两种场景的统一视角

```
场景 A:  research.md (AI单向输出)  →  plan.md (标注循环)  →  实现
场景 B:  research.md (标注循环)    →  plan.md (标注循环)  →  实现
```

唯一的区别是 research.md 是否需要标注循环。取决于人的意图是否已经清晰。

**标注循环是同一个协议，作用在不同的文档上。**

### 标注协议详细设计

#### 反馈模型

反馈模型直接从"人需要向 AI 传递什么信息"推导：

| 人要表达的 | 推荐写法 | AI 的回应方式 |
|-----------|---------|-------------|
| 我知道你不知道的事 | 自由文本补充上下文 | 纳入信息，更新文档，说明如何影响了结论 |
| 我不理解你说的 | 自由文本提问 | 用证据解释。如果解释不了，承认并标记为风险 |
| 你的结论/方案有误 | 自由文本指出问题或要求修改 | 评估改动影响。如果人说的对 → 改。如果有问题 → 用证据说明问题，给替代方案，让人决定 |
| 你的研究不够深 / 你遗漏了重要内容 | 自由文本指出深度不足或遗漏 | 继续调查指定方向，补充到文档 |
| 这里需要更多调查才能决定 | `[PAUSE]` + 暂停原因 | 暂停当前文档的修改，去补充研究，将结果写回 |

除了 `[PAUSE]` 以外，不要求固定 marker。AI 需要从反馈内容中推断意图，并把这个推断记录到 Annotation Log。

#### AI 回应的核心原则

**人不一定对，AI 不能盲从。**

```
如果人的标注是正确的：
  → AI 采纳，更新文档，说明改了什么

如果人的标注是错误的或有风险的：
  → AI 用证据（file:line 引用、代码事实）说明问题
  → AI 提供替代方案
  → AI 让人做最终决定
  → 无论人怎么决定，记录在 Annotation Log 中（包括 AI 的反对意见）
```

这不是"AI 挑战人的权威"，而是"AI 有责任把它知道的信息传递给人"。
共同理解要求双向信息流。如果 AI 知道人错了但不说，共同理解就是假的。

**但最终决定权始终在人手上。**

#### Annotation Log

每轮标注和回应记录在文档末尾的 `## Annotation Log` 中：

```markdown
## Annotation Log

### Round 1 (2026-03-02)

**Question § Execution Paths**
"ServiceB.process() 为什么标记为安全？它不是没做空值检查吗？"
→ 推断意图：质疑研究结论
→ 你说得对，这是研究结论的错误。ServiceB.process() 确实不检查 null 输入（src/b.ts:122）。
  已修正 Risks 段：从 ✅ 改为 ❌，并在 plan 的风险段中新增应对措施。

**Change Request § D1**
"改用 Redis 做缓存"
→ 推断意图：要求修改缓存设计
→ ⚠️ 当前架构使用进程内缓存（evidence: 全项目 0 个 Redis 依赖）。
  引入 Redis 的影响：(1) 新增基础设施依赖 (2) 同步→异步改造 (3) 序列化开销。
  替代方案：现有 CacheManager 加 TTL（src/cache.ts:30）。
→ 等待你的决定。

**Depth Issue § Execution Paths**
"EventBus 的 listener 还没追"
→ 推断意图：指出研究深度不足
→ 已追踪：ServiceB.process() → EventBus.emit("user.updated") → 3 个 listener:
  · AuditListener (src/listeners/audit.ts:15) — 纯日志，无副作用
  · CacheListener (src/listeners/cache.ts:30) — 清缓存，有时序风险
  · NotifyListener (src/listeners/notify.ts:8) — 异步通知，无阻塞风险
  已更新 Execution Paths 段。
```

### research.md 的设计

research.md 是"AI 的理解"的载体。它的目的是让人能快速判断：
- AI 是否充分理解了代码
- AI 的理解是否正确
- 是否有遗漏

**不设固定模板。** 因为不同任务需要不同的研究结构。但 AI 的研究应该回答以下问题：

1. **研究了什么** — 范围、读了哪些文件
2. **代码怎么运行的** — 关键路径、调用链（必须有 file:line 引用）
3. **有什么风险** — 已确认的、已排除的、待确认的
4. **还不知道什么** — 没读的文件、没验证的假设

如果需要引导 AI 做更深入的研究，通过标注循环实现，而不是通过模板检查。

**模板检查的问题**：AI 可以形式上填满所有段落但实质浅薄。
**标注循环的优势**：人看了 research 后可以用自由文本精准指出不足，必要时用 `[PAUSE]` 中断当前方向先补调查。

### plan.md 的设计

plan.md 是"双方共识"的载体。它应该让人能判断：
- AI 打算做什么（具体改动）
- 为什么这么做（理由、取舍）
- 有什么风险（可能出问题的地方 + 应对）

**同样不设固定模板。** 但 AI 的计划应该包含：

1. **做什么** — 具体改动，引用 research 中的发现
2. **为什么** — 设计理由、考虑过的替代方案、为什么选这个
3. **影响什么** — 改动涉及的文件、受影响的调用方
4. **风险** — 可能出问题的点、应对策略

**plan.md 不包含 todolist。** Todolist 在 plan 审批后由 AI 生成。
原因：标注循环中如果改了设计方向，之前的 todolist 全部作废。分离可以避免大量无效改动。

### 写锁的设计

写锁的职责是**保证时序**，不是质量检查。

```
判断逻辑（与当前 baton 几乎相同）：

1. 目标是 .md 文件 → 放行（文档写入永远不锁）
2. 找 plan.md
   · 不存在 → 阻止 + 提示"先完成研究和计划"
   · 存在但无 <!-- BATON:GO --> → 阻止 + 提示"完成标注循环后审批"
   · 存在且有 <!-- BATON:GO --> → 放行
```

**不做复杂的质量门控。** 写锁不检查 research.md 是否"够深"、plan.md 是否"够完整"。
这些由标注循环保证 — 人不满意就不会加 `<!-- BATON:GO -->`。

**写锁的简洁是刻意的设计决策：**
- 简单 → 不容易出 bug → 可靠
- 简单 → token 开销小
- 简单 → 与任何外部系统兼容（不需要解析复杂格式）

### 阶段引导（phase-guide.sh）

SessionStart 时检测当前状态，提供上下文提示：

```
状态 1：无研究文件（research.md 或 research-<topic>.md），无计划文件
→ 提示：研究阶段。深入阅读代码，产出研究文件。
  追踪调用链到底层，标记风险（✅ 已确认安全 / ❌ 发现问题 / ❓ 待确认）。
  每个结论附 file:line 证据。

状态 2：有研究文件，无计划文件（plan.md 或 plan-<topic>.md）
→ 提示：计划阶段。基于研究文件和需求产出计划文件。
  包含：做什么、为什么、影响范围、风险应对。
  不要写 todolist — 等人审批后再生成。

状态 3：有计划文件，无 BATON:GO
→ 提示：标注循环阶段。等待人的标注。
  收到标注后逐条回应，记录到 Annotation Log。
  人的标注不一定正确 — 如有问题用证据说明，给替代方案。

状态 4：有计划文件 + BATON:GO
→ 提示：实现阶段。按 todolist 顺序实现。
  每个 item 完成后 typecheck/test，通过后标记 [x]。
```

## 六、这个设计如何解决最初的四个问题

| 问题 | 根因 | 本设计如何解决 |
|------|------|---------------|
| 研究太浅 | 没有深度标准，AI 不知道什么叫"够深" | 阶段引导提示追踪调用链、标记风险、附 file:line。但真正保障深度的是标注循环 — 人用自由文本指出深度不足或遗漏，必要时用 `[PAUSE]` 要求先补调查 |
| 研究没进 plan | research→plan 过渡是可选的 | plan 的设计要求引用 research 发现。阶段引导提示"基于 research 产出 plan" |
| Plan 太粗无法标注 | 无标注格式和流程 | 标注协议提供 free-text + `[PAUSE]` 模型、AI 回应规则和 Annotation Log |
| 与 superpowers 打架 | 两个系统争抢生命周期控制 | Baton 只做三件事（文档+标注+写锁），不涉及方法论。方法论由 superpowers 或其他系统提供 |

## 七、Baton 的职责边界

```
Baton 做什么：
├── 写锁 — 时序保证（BATON:GO + ## Todo 才能写源码）
├── 流程阶段检测 — 结构性检查（流程步骤是否完成），非质量判断
├── 标注协议 — free-text + `[PAUSE]` + AI 回应规则
├── Annotation Log — 持久化每轮对话
├── 阶段引导 — 当前阶段的行为提示
└── research/plan 深度引导 — 提示（非强制）应包含的内容

Baton 不做什么：
├── 不判断文档质量 — "research 够不够深" 由人通过标注循环判断
├── 不做项目管理 — 不跟踪任务分配、优先级、截止日期
├── 不规定方法论 — 研究/实现/代码审查方法由其他系统决定
```

**判断原则：如果一个功能不直接服务于"构建共同理解"，就不属于 Baton。**

## 八、与 design-comparison.md 中方案的关系

| 方案 | 本设计吸收了什么 | 本设计舍弃了什么 |
|------|----------------|----------------|
| A 当前 Baton | 写锁机制、极简哲学 | ~~极简到引导不足~~ |
| B Baton_bak v2 | Annotation Log、研究回环思路 | ~~8 个 skill、CLI、三层治理、Context Slice、显式 marker 模型~~ |
| C Superpowers | 不吸收（正交系统，各管各的） | — |
| D Planning-with-Files | 不吸收（不同层次的问题） | — |
| E 协议层 | "定义标准不管方法"的思路 | ~~质量门控、模板检查~~ |
| F 条目级门控 | — | ~~条目级 [GO]、文件范围锁~~ |

**关键舍弃**：
- **质量门控**：用标注循环替代。人的判断比 hook 的格式检查更准。
- **Context Slice**：交给 superpowers 的 subagent 模式，或者不用。
- **条目级 [GO]**：增加了复杂度但没解决核心问题（共同理解）。写锁保持简单。
- **固定模板**：引导代替强制。AI 应该根据任务自主组织文档结构。

## 九、设计决策记录

### 已决定

**Q1: Todolist 生成后发现问题怎么办？**
→ 人删除 `<!-- BATON:GO -->`，写锁重新生效。回到标注循环，修改 plan 后重新生成 todolist。
删除 BATON:GO 即"回退"，这和当前 baton 的 re-lock 机制一致 — 简单、无歧义。

**Q2: 实现过程中发现计划遗漏怎么办？**
→ AI 停下来，更新 plan.md，人 review 后继续。
如果问题严重到需要改设计方向 → 人删除 BATON:GO，完整回退到标注循环。
如果只是小的补充 → AI 在 plan.md 追加说明，人确认后继续（不需要删 BATON:GO）。

**Q3: research.md 是否总是必要？**
→ 简单改动（1-2 个文件、改动明确）允许跳过 research.md，直接写 plan.md。
plan.md 的 Context 段可以内嵌简要的研究内容。
写锁不依赖 research.md 存在（只检查 plan.md + BATON:GO）。

**Q4: 标注循环的退出条件？**
→ 人说"生成 todolist"或"开始实现"即为退出。不需要额外标记。
BATON:GO 是唯一的审批标记，由人添加。

### Subagent 与依赖关系（Q5）

有依赖关系的 todolist items 使用 subagent **有问题**：

```
Item #1: 重构 UserService.validate()
Item #2: 更新 AuthHandler 调用新的 validate()  ← 依赖 #1 的实际输出
Item #3: 新增 AdminHandler 复用 validate()      ← 依赖 #1 的实际输出
```

如果 #1 由 subagent A 实现，#2 由 subagent B 实现：
- subagent B 只知道计划说 "#1 会重构 validate()"
- 但不知道 subagent A **实际**把 validate() 的签名改成了什么
- 如果 A 的实现和计划有微小偏差（比如参数顺序变了），B 会基于错误假设工作

**正确的处理方式**：

```
有依赖关系的 items → 同一会话顺序执行（每个 item 能看到前一个的实际代码）
无依赖关系的 items → 可以 subagent 并行

判断依赖：
- item B 的 Files 和 item A 的 Files 有交集 → 有依赖
- item B 的改动引用了 item A 修改的函数/接口 → 有依赖
- 其他情况 → 无依赖，可并行
```

**这不是 Baton 要强制的事。** Baton 管的是"共同理解"，不管"执行策略"。
这属于实现阶段的指导建议，写在 workflow 引导中即可。

### 任务完成后的文档生命周期（Q6）

任务完成后，plan 需要标记为完成，否则：
- 下次新任务会被旧文档干扰
- phase-guide.sh 检测到旧 plan 会给出错误的阶段提示
- Annotation Log 中的决策记录会丢失

**完成流程**：

```
任务完成后：
1. 在 plan 文件中添加 <!-- BATON:COMPLETE --> 标记（独立一行）
   → 标记后的 plan 对 parser 不可见，无需移动文件

2. 完成后的 plan 和 research 留在 baton-tasks/<topic>/ 目录中
   → 下次 SessionStart 时 phase-guide.sh 检测到无活跃 plan，提示研究阶段
```

**完成标记的价值**：
- plan 中的 Annotation Log 记录了**为什么做了这些决策**，是未来的参考
- research 记录了**代码在那个时间点的结构**，对同区域的后续改动有参考价值
- 如果后续任务涉及同一区域，AI 可以读 baton-tasks/ 中的文件快速恢复上下文

**完成时机**：
- 所有 todolist items 标记 [x] 且测试通过后
- phase-guide.sh 检测到"plan 有 BATON:GO + 所有 items 已完成"时，提示完成流程

**stop-guard.sh 的角色**（继承自当前 baton）：
- 如果会话中途停止且有未完成的 items → 提醒有未完成的工作
- 如果所有 items 已完成但未标记完成 → 提示添加 BATON:COMPLETE

## 十、剩余开放问题

1. **对超长 todolist (15+ items) 的处理？**
   Baton 不做 Context Slice。建议（写在引导中，不强制）：
   - 无依赖 items → subagent 并行
   - 有依赖 items → 同一会话顺序执行
   - 超长任务 → 考虑分批（先实现前 5 个，typecheck 后再继续）

2. **多任务并行？**
   如果同时做两个不同任务，plan.md 会冲突。
   继承当前 baton 的方案：`BATON_PLAN=plan-auth.md` + git worktree。
   `BATON_PLAN` 有两个用途：(1) 使用 topic-named 文件（如 `plan-auth.md`）；(2) 同一工作区存在多个计划文件时，明确告知写锁应检查哪个文件。
   每个 worktree 有独立的研究文件和计划文件。

3. **Baton 是否应该作为 Claude Code 的 skill/plugin 发布？**
   当前 baton 是 hook 脚本 + workflow.md。
   如果作为 skill 发布，标注协议和 AI 回应规则可以更自然地注入到 AI 的 context 中。
   但 skill 无法提供物理写锁（需要 hook）。
   可能的方案：hook（写锁）+ skill（标注协议引导）双层发布。
